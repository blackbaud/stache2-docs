Stache wrapper

The {{ stache.jsonData.global.productNameShort }} wrapper is the primary element in a {{ stache.jsonData.global.productNameShort }} project. It contains most of the other {{ stache.jsonData.global.productNameShort }} tags and content. Most page-level configuration is handled by the {{ stache.jsonData.global.productNameShort }} wrapper, including layouts and navigation. Typically, this is the first tag to add to any {{ stache.jsonData.global.productNameShort }} page after the app-nav tag. If your SPA is using the omnibar instead of app-nav, this should be the very first tag on the page.

{{ stache.jsonData.global.productNameShort }} takes a prescriptive approach to configuration. Most properties on the stache tag have defaults, so you rarely need to set them yourself. The main exceptions are the pageTitle, navTitle, and navOrder.

// Your content here.

Application navigation For SPAs created by Blackbaud staff, we recommend the omnibar instead of the app-nav component. To learn how to set up top-level navigation with the omnibar, see step 4 of the Blackbaud staff Create a New Stache SPA tutorial.

To provide top-level navigation within your app, you can use the app-nav component. This shared component from the SKY UX default template enables you to display a navigation bar at the top of pages.

Display navigation bar

To display a navigation bar at the top of a page, place the app-nav tags at the start of the page's index.html file. To include the navigation bar throughout your SPA, start every page with the app-nav tags.

<app-nav></app-nav> // Your content here.

Set up navigation bar

To display entries in the navigation bar, you must manually configure the top-level navigation for your SPA. To use localization strings for the top-level navigation items, see the SKY UX Create a page tutorial .

  1. In the src/app/shared folder, open the app-nav.component.ts file. You need entries in the AppNavComponent class for each top-level navigation item.

  2. Use the name property to specify a label for a navigation item.

    If you specify a localization string from resources_en_US.json, the navigation bar displays the message value from that localization string.

  3. Use the path property to specify where to direct users when they click a navigation item. When you specify the path to a folder in your SPA, the navigation bar opens the index.html file in that folder.

    4. export class AppNavComponent { public nav = [ { name: 'app_nav_home', path: '/' }, { name: 'app_nav_about', path: '/about' }, { name: 'Demo', path: '/demo' } ]; }
Custom CSS

In {{ stache.jsonData.global.productNameLong }}, you can add custom CSS (or SCSS) through SCSS files. We recommend you create a file named app-styles.scss and place it in the src/app folder. For those who don't know SCSS, you can use CSS inside an SCSS file. For example:

//CSS img { border: 1px solid #cccccc; }

After the app.scss file is in place, you need to tell {{ stache.jsonData.global.productNameLong }} where to find your custom CSS. From the src/app folder, open the skyuxconfig.json file. Under the app section, add the styles property to reference your new SCSS file.

"app": { "title": "My Stache Site", "styles": [ "src/app/app-styles.scss" ] } Command line

Azure DevOps

Unless you are working locally, all new Blackbaud RDO staff SPAs are set up as Git repos in Azure DevOps . Azure DevOps has pull requests just like GitHub. However, there are a few key differences to know about.

  1. Each time you create a PR, or make a change to your pull request, it kicks off a build. The build takes a few minutes to complete and tests to make sure there aren't any problems with your code before it merges.
  2. The PR build is one part of the PR approval process. Most Azure DevOps repos also require you to get at least one code review approval.

Because of these differences, each change to your repo will take, at minimum, a few minutes to release to your live SPA.

You can use tools such as VS Code to work with a local clone of your Stache site, but the web-based editor in Azure DevOps lets you make quick changes without leaving the browser. To learn how to make changes with the Azure DevOps web-based editor, see the Edit Stache online tutorial.

Branch Policies and Security

Stache SPAs in Azure DevOps are set up with standard policies and security. Learn how to edit these policies for your documentation sites.

Standard policies for a pull request into the master branch include:

  • Pull request approval by at least 1 reviewer
  • Successful pull request build
  • Person who created the SPA is configured as a "required" reviewer
