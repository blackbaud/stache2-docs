Stache welcomes contributions from anyone. In order to contribute, a better understanding of Stache and how it functions as a whole is required. Stache is comprised of two main pieces: components and plugins.

Components

Stache components are the primary vechicle for consuming Stache. These components come in two flavors: content creation and structural. The content creation components are components consumed directly by users to author content. These are the components listed on the content page.

Structural components serve a more behind the scenes purpose. They facilitate and enhance the content creation components by abstracting away a lot of complexity. These components are not meant to be used directly for content creation. Many of these structural components are combined to create layouts, which serve as the skeleton for Stache pages. For example, the sidebar layout is comprised of the stache-sidebar, stache-layout, stache-breadcrumbs, stache-page-header, and stache-table-of-contents components.

When creating new components, it is important to consider this distinction. Will this component be used directly by consumers to create content? Or is this more of a configuration or structural component that is better used in a layout? For example, a content creator would create a block quote, but would never manually generate a table of contents.

Plugins

The other crucial piece of Stache is the plugin. Plugins serve to modify and manipulate components, services, and configuration at build time and are vital to many core features of Stache. Many of the global configuration settings and data services, such as the Stache Data Service and the Stache Http Service are compiled and injected through plugins during build. Some components even require build time compilation to overcome run time limitations, such as the stache-include and the stache-code-block components.

Plugins live in a separate code base outside of the core Stache project. You can read more about the plugin framework in the repo's README.
